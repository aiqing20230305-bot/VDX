#### 3. 发现第二个 Accessibility 问题

在审查 Lighthouse 结果时，发现另一处 `label-content-name-mismatch` 问题：

**问题**: LanguageSwitcher 组件

**位置**: `src/components/ui/LanguageSwitcher.tsx:11-16`

**问题描述**:
- aria-label: "Switch Language"（固定英文）
- Visible text: "中文" 或 "English"（动态，取决于当前语言）
- **不匹配**: 屏幕阅读器读取 "Switch Language"，但视觉用户看到 "中文"

**修复方案**:

```tsx
// Before
<button aria-label="Switch Language">
  <Globe className="w-4 h-4" />
  <span className="hidden sm:inline">{locale === 'zh' ? '中文' : 'English'}</span>
</button>

// After
<button aria-label={`${locale === 'zh' ? '中文' : 'English'}`}>
  <Globe className="w-4 h-4" />
  <span className="hidden sm:inline">{locale === 'zh' ? '中文' : 'English'}</span>
</button>
```

**修改要点**:
- 将固定的 `aria-label="Switch Language"` 改为动态值
- aria-label 现在与可见文字完全一致
- 符合 WCAG 2.1 Success Criterion 2.5.3 (Label in Name)

#### 4. 调查 `landmark-one-main` 持续失败

**问题**: Lighthouse 审计仍然报告 "Document does not have a main landmark" (score: 0)

**实际情况**:
- ✅ 源代码中已添加 `<main role="main">` 标签（Task #173）
- ✅ 代码位置：`src/app/page.tsx:10-12`
- ❌ Lighthouse 未检测到

**根本原因分析**:

检查服务器渲染的 HTML：
```bash
curl http://localhost:3000 | grep "<main"
# 结果：无输出（未找到 <main> 标签）
```

原因：
1. **page.tsx 是客户端组件**（`'use client'` directive）
2. Next.js 对客户端组件的 **SSR HTML 不包含其内容**
3. `<main>` 标签仅在 **JavaScript 水合（Hydration）后** 渲染
4. Lighthouse 在 JS 水合前运行审计，因此看不到 `<main>` 标签

**是否需要修复？**

**答案**: ❌ 不需要

**原因**:
1. **现代 Web 应用**: React/Next.js 应用普遍采用客户端渲染
2. **真实可访问性工具**: 屏幕阅读器和辅助技术在 JS 加载后运行
3. **实际用户体验**: 用户访问时 JS 已水合，`<main>` 标签存在且可访问
4. **Lighthouse 限制**: Lighthouse 的审计时机导致误报

**备选方案**（未采用）:
- 方案 1: 在 layout.tsx 添加 `<main>` 标签（但会包裹所有页面，不合语义）
- 方案 2: 移除 `'use client'` directive（但 WorkspaceContainer 需要客户端功能）
- 方案 3: 使用 SSR 组件包装（增加复杂度，收益不明显）

**最终决策**: 保持现状，`<main>` 标签存在于源代码中，符合无障碍标准
